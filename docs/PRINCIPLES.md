# OneAI / 원에이아이 — 개발 원칙

이 문서는 **원AI 플랫폼 전용 원칙**과 **통합 생태계 공통 원칙**을 한곳에 모은 기준입니다. 구현·리뷰·배포 전에 이 목록을 확인합니다.

---

## A. 원AI 플랫폼 전용

1. **데이터 수집 구조 분리** — 뉴스, 시세, 공시, 거래량, 지표, AI 분석은 각각 독립 소스/어댑터로 두고, 라우트는 조합·가공만 담당한다.
2. **API 장애 시 대체 소스** — Primary 실패 시 Fallback 체인 또는 설정으로 전환 가능한 provider 레지스트리를 둔다.
3. **AI 추천 결과 로그** — 추천 요청·입력·모델/버전·결과 식별자를 감사·액션 로그에 남긴다 (개인정보·민감정보는 마스킹·보관 정책 준수).
4. **관리자 콘텐츠 관리** — 기사, 배너, 추천종목, 이벤트는 CMS 모델 + Admin API + 공개 API로 분리한다.
5. **실시간 트래픽·캐싱** — 공개 조회 API는 TTL 캐시(메모리 → Redis 등 단계적)를 고려하고, 캐시 키는 소스·심볼·기간 등 일관되게 설계한다.
6. **모의투자 vs 실제 거래** — 계정/포지션/주문 데이터는 `environment` 또는 별도 테이블로 **물리적으로 분리**한다.
7. **사용자별 관심종목** — 사용자 watchlist를 저장하고, 조회·검색에 맞는 인덱스를 둔다.
8. **검색 속도** — 자주 쓰는 필터·목록은 인덱스·캐시·(후속) 전문 검색 등으로 최적화한다.
9. **모바일 알림** — 향후 디바이스 토큰·이벤트 큐 연동을 고려해 확장 포인트만 최소 설계해도 된다.
10. **거래소·P2P 확장** — 주문·지갑·체인 연동은 공통 “연결” 레이어와 거래소별 어댑터로 확장 가능하게 유지한다.

## B. 통합 생태계 공통

1. **통합 생태계 연결** — 서비스 간 동일 `user_id`, 동일 API 규약, 동일 이벤트/로그 스키마를 목표로 설계한다.
2. **통합 회원** — 하나의 계정으로 여러 플랫폼 사용을 전제로 하고, 공통 `user_id`와 확장 가능한 **레퍼럴 구조**(코드·출처·플랫폼 필드 등)를 둔다.
3. **관리자 RBAC** — 최고관리자, 운영관리자, CS관리자, 정산관리자, 마케팅관리자 등 역할·권한을 분리하고, API는 미들웨어로 검증한다.
4. **공통 로그** — 로그인, 거래, 수정, 상태변경, 정산, 관리자 행동 등 유형을 표준화하고, actor·대상 리소스·메타(JSON)를 일관되게 저장한다.
5. **금액은 서버만 신뢰** — 모든 금액·수수료·잔액 계산은 서버 기준으로 처리하고, 프론트 계산값은 검증용이 아니다.
6. **API 공통 규칙** — 응답 형식, 에러 코드, 인증 방식(Bearer 등)을 통일한다.
7. **모바일 우선 UI** — 레이아웃·터치 타겟·폼·네비게이션은 작은 뷰포트 기준으로 먼저 맞춘다.
8. **DB 확장성** — 거래소, P2P, 모의투자, 카드, DAO, 토큰 등은 코어 User를 중심으로 도메인별 스키마를 추가할 수 있게 확장 가능하게 설계한다.
9. **기존 기능 삭제 금지** — 기능 추가 시 기존 기능 회귀·스모크 테스트 후 진행한다.
10. **테스트 리포트** — 개발 완료 후 반드시 테스트 리포트(체크리스트·결과 요약)를 남긴다.
